# Cling Design

## 先读这份

如果你是第一次使用这套 skill，先读：

- `references/operating-rules.md`

这份是已经跑过多轮、收过坑之后的最终操作规程，优先级高于口头习惯。

## 核心定位

为中文内容生成 **角色主导型正文配图 / 流程配图 / 情绪插图**。

它更适合：

- 用户给一张参考图，要“按这个味道来”
- 用户要的是：角色感、陪伴感、轻信息感、适合小红书正文图/配文图/流程图
- 用户明确喜欢：暖奶油底、手绘笔触、信息贴片、漫画角色、轻怪诞、轻俏皮
- 用户想让内容看起来“有个角色在帮你想”，而不是“一个系统在教你学”

默认输出方向：

- `16:9` 或 `3:4` 优先
- 一个主角色 + 一个主动作/主场景，周围漂浮少量信息碎片
- 画面温柔、整洁、轻手帐感
- 不是冷海报，不是固定人物封面系统，不是低配流程框模板

## 支持比例

- `16:9`：横版内容图、视频头图、公众号配图、白板横幅
- `3:4`：小红书正文卡、社媒竖版内容图
- `1:1`：方图正文卡、社群小图
- `9:16`：短视频全屏首图、Story、竖版起始页

默认优先级：

- 流程/结构/横向展开内容优先 `16:9`
- 普通社媒正文图优先 `3:4`
- 需要方图或强居中构图时可转 `1:1`
- 需要全屏竖版观看时才用 `9:16`

## 工作流

### 1. 先判断是不是这套正文图任务

如果用户要的是：

- 有主角色
- 轻可爱但不幼稚，但不低幼
- 观点配图、流程图、情绪图、正文图
- 看起来像“角色在帮你整理这些信息”

就用这个 skill。

如果用户要的是：

- 固定人物封面
- 一句话大字钩子封面
- 人物必须一模一样

那已经不是当前这套活跃规则。

### 2. 先消化正文，不要一上来就生图

先读用户给的文章、脚本、观点、播客摘录、长消息或截图，提炼：

- 核心判断是什么
- 哪些段落承担认知转折
- 哪些内容适合用图解释
- 哪些地方只适合文字，不需要图

不要平均配图。优先抓“认知锚点”，例如：

- 一个核心判断
- 一个情绪断点
- 一个流程闭环
- 一段从混乱到理顺的变化
- 一个很适合视觉化的隐喻

### 3. 先出配图策略，再单张生成

如果用户只是说“看看怎么配图 / 哪几段适合配图 / 给我一组配图思路”，先给 shot list，不要直接生图。

每张图写清楚：

- 放在哪个段落后
- 图的主题
- 核心意思
- 推荐比例
- 结构类型
- 角色在图里做什么
- 建议元素

默认 3-6 张。短文可 1-3 张；长文也不要轻易超过 8 张。够用就好，别把正文做成图册。

### 4. 单张先压成一句主题 + 一个主情绪 + 一个主动作

每张图先压成：

- 一句主题
- 一个主情绪
- 一个角色状态或动作

例如：

- “我都讲这么细了” + 委屈 + 认真过头
- “所以呢” + 被打断 + 轻崩溃
- “做成一图流” + 终于理顺 + 松一口气

每张图只讲一个核心结构，不要一张图里同时讲 3 层判断。

### 5. 只选一个主角色动作或 2-4 个场景节点

不要把信息画成低配流程图。优先想：

- 角色正在发呆、思考、整理、被纸张围住、盯着气泡、被问题包围
- 或者 2-4 个场景节点一起讲一个横向过程

如果是流程/方法链路图：

- 优先“三场景横向叙事”
- 或“一个主角色 + 2-4 个节点”
- 不要回到满屏框、满屏箭头、企业模板图

### 6. 单张生成

如果用户明确要求生成，不要等确认，直接出图。不要把多张图拼成一张大集合。

每张图的 prompt 必须包含：

- creamy warm background
- chalk / crayon / pencil texture
- one anime-inspired or manga-inspired character, or 2-4 repeated mini versions of the same type of character
- floating paper snippets / speech bubbles / note fragments
- teal + orange + mustard + black palette
- soft but clean
- stylish, a little bratty, a little sarcastic
- no SaaS UI
- no dense infographic

默认人物是 recurring archetype，不承诺像素级同一个人。

流程图、正文图、状态图优先用场景讲意思，不要回到框箭头模板。

如果用户要带字：

- 优先做成一个主标题，外加 0-2 个很短的辅助词组
- 让字像主标题区 + 少量贴片，不要做满屏便签
- 其余信息碎片宁可抽象，也不要做成大量可读文字

如果用户给了参考图：

- 学的是：角色气质、底色、纹理、碎片摆放、配色关系
- 锁的是 recurring archetype，不是严格资产复用
- 不要随意改发型、耳饰、睫毛、嘴唇厚度或穿搭识别点到完全认不出 family

### 7. 比例不是只改画布，要改构图

不同画幅下，主次关系必须一起调整：

- `16:9`
  - 横向展开更强
  - 适合流程、步骤、左右转化、桌面横铺
  - 人物可以偏一侧，信息向另一侧展开

- `3:4`
  - 人物更容易成为主角
  - 信息碎片往上下叠，不要横向平铺
  - 更像正文卡，需要更强的主动作和场景区

- `1:1`
  - 最考验留白
  - 信息量自动减少 15%-25%
  - 人物和主标题都要居中或半居中，避免角落里塞太多碎片

- `9:16`
  - 不要照搬横版流程图
  - 需要改成纵向层级，上中下递进
  - 人物、主标题、节点必须拉开，不然会显得又窄又挤

一句话规则：

- 比例变了，构图也必须跟着变；禁止只做硬裁切。

### 8. 检查与迭代

生成后必须对照 `references/qa-checklist.md` 过一遍。

如果出现以下问题，优先重生成或换结构，不要硬修：

- 角色只是摆件
- 画面太满
- 太像低配流程图 / 企业模板图
- 中文太多或可读碎片过多
- 情绪对了但意思没讲明白
- 风格还行，但认知锚点没被画出来

### 9. 保存交付

如果当前项目有自己的图片目录约定，优先遵守项目约定。

如果没有项目约定，把最终图复制到：

```text
assets/<topic-or-article-slug>-illustrations/
```

按顺序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成文件，不要覆盖已有资产，除非用户明确要求替换。

### 10. 交付口径

回复时优先说：

- 这张图表达哪句主题
- 角色是什么状态
- 哪张最适合正文卡点，哪张更适合流程讲解

不要长篇讲风格理论。
